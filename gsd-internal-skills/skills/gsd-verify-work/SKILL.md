# gsd-verify-work

## 作用

用于对已完成的工作做自动验证和人工验收引导，并把验证结果转化为可回流的返工建议。

它必须体现：

- 自动验证
- 人工验收引导
- 验证结果记录
- 回退 / 返工建议

## 适用场景

- phase 已执行完成，需要确认交付质量
- 需要组织一轮回归或人工确认
- 需要把问题沉淀成后续修复输入

## 不适用场景

- 工作尚未进入可验证状态
- 还没有可参考的执行结果或任务记录
- 当前首先需要做的是根因定位

## 输入要求

建议具备：

- `PLAN.md`
- `TASKS.md`
- `STATE.md`
- 执行过程中的变更说明

## 输出要求

至少输出或更新：

- `VERIFY.md`
- `STATE.md`

验证结果应明确区分：

- 已通过
- 未通过
- 待人工确认
- 待返工

## 建议读取的状态/模板文件

- `.planning/PLAN.md`
- `.planning/TASKS.md`
- `.planning/STATE.md`
- `.planning/VERIFY.md`
- `templates/VERIFY.md`

## 工作步骤

### 1. 明确验证对象

确认本次验证的是：

- 整个 phase
- 某组任务
- 某个缺陷修复

### 2. 做自动验证

优先检查：

- 关键功能路径
- 关键接口和依赖
- 与计划中承诺的验收点是否一致
- 是否存在明显回归

### 3. 引导人工验收

对于自动验证无法覆盖的部分，应明确告诉人工协作者：

- 要看什么
- 怎么看
- 预期结果是什么
- 发现问题后如何反馈

### 4. 记录验证结果

在 `VERIFY.md` 中记录：

- 验证范围
- 验证步骤
- 结果
- 发现的问题
- 严重级别

### 5. 给出回退或返工建议

如果失败，给出：

- 是否能走 `gsd-fast`
- 是否需要重新 `gsd-plan-phase`
- 是否先进入 `gsd-debug`

## 约束与注意事项

- 不要只记录“通过/不通过”，要写清证据
- 人工验收项要尽量具体，不要给模糊指令
- 发现问题后必须给出下一步建议，而不是停在结论
- 验证要尽量回到原计划和原验收标准

## 与其他 skills 的衔接关系

- 上游通常是 `gsd-execute-phase`
- 发现小问题可转 `gsd-fast`
- 发现结构性问题可回到 `gsd-plan-phase`
- 根因不明的问题可转 `gsd-debug`
