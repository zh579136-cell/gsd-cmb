# gsd-discuss-phase

## 作用

用于某个 phase 的边界澄清与上下文锁定。它的目标不是直接做实现，而是把会影响方案和结果的灰区变成已知决策。

典型产出：

- `CONTEXT.md`

## 适用场景

- 已知道要做一个 phase，但具体实现边界不清
- 需要先确认关键决策、限制和偏好
- 需要把参考资料、既有模式、用户偏好沉淀下来

## 不适用场景

- 需求还停留在项目级定义，尚未切到 phase
- 已经有充分上下文且无需再澄清
- 只是非常小的改动

## 输入要求

最好具备：

- 当前 phase 的目标
- 上游 roadmap 或需求
- 既有代码库背景
- 用户明确偏好或限制

## 输出要求

`CONTEXT.md` 至少应包含：

- phase 边界
- 已锁定的决策
- 参考资料
- 代码上下文
- 具体偏好
- 延后项

## 建议读取的状态/模板文件

- `.planning/ROADMAP.md`
- `.planning/PROJECT.md`
- `.planning/REQUIREMENTS.md`
- `.planning/CODEBASE_MAP.md`
- `templates/CONTEXT.md`

## 工作步骤

### 1. 读取 phase 目标

先从 roadmap 和需求中确认：

- 当前 phase 到底要交付什么
- 哪些内容不属于这个 phase

### 2. 找出灰区

灰区通常包括：

- 交互与表现方式
- 数据流或接口边界
- 错误处理预期
- 与既有系统的融合方式
- 需要遵守的技术或合规约束

### 3. 选择讨论模式

可使用两种模式：

- 访谈式：适合新领域、用户有明确偏好时
- 假设式：适合老系统、代码模式清晰时

### 4. 锁定决策

对每个灰区，尽量形成以下结论：

- 明确做法
- 明确不做法
- 允许 agent 自主裁量的范围
- 延后处理的内容

### 5. 写入 CONTEXT

要求让后续规划者能直接消费，不必再重复问同样的问题。

## 约束与注意事项

- 不要在 discuss 阶段引入新的 scope
- 不要把执行方案细节写得过深
- 延后项要明确记录，避免后续又被重新拉入当前 phase
- 没有明确证据的系统假设要标记为“待确认”

## 与其他 skills 的衔接关系

- discuss 完成后进入 `gsd-plan-phase`
- 如果发现其实缺少代码背景，回到 `gsd-map-codebase`
- 如果问题本质不是规划而是故障，转 `gsd-debug`
