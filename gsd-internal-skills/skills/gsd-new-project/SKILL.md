# gsd-new-project

## 作用

用于新项目或新需求初始化，帮助内网 agent 把一个模糊目标沉淀为可持续推进的项目定义。

核心产出：

- `PROJECT.md`
- `REQUIREMENTS.md`
- `ROADMAP.md`
- `STATE.md`

## 适用场景

- 从 0 开始的新项目
- 老项目里的一条新需求线，需要独立形成规划
- 用户只有想法、背景和约束，还没有清晰拆分

## 不适用场景

- 已经有成熟 `.planning/` 文件且只是在执行某个 phase
- 只需要分析既有系统，不涉及新规划
- 只是非常小的增量需求

## 输入要求

至少应有以下信息中的一部分：

- 项目目标或问题陈述
- 目标用户或使用场景
- 范围边界
- 关键约束
- 参考资料或上游文档

## 输出要求

至少输出：

- 项目定义
- 当前范围
- 非目标
- phase 切分建议
- 当前状态初始化

## 建议读取的状态/模板文件

- `templates/PROJECT.md`
- `templates/REQUIREMENTS.md`
- `templates/ROADMAP.md`
- `templates/STATE.md`

如果是老仓库里的新需求，也应先看：

- `.planning/CODEBASE_MAP.md`
- `.planning/PROJECT.md`

## 工作步骤

### 1. 识别初始化对象

判断是在：

- 初始化一个新项目
- 还是在老项目里初始化一条新需求线

### 2. 收敛目标

围绕以下问题收敛：

- 要解决什么问题
- 谁来用
- 第一版必须做到什么
- 哪些内容不是当前范围
- 成功的最低标准是什么

### 3. 收敛约束

重点确认：

- 技术约束
- 组织约束
- 时间约束
- 依赖系统
- 数据和安全限制

### 4. 形成项目级文档

写入：

- `PROJECT.md`
- `REQUIREMENTS.md`

要求：

- 需求可执行
- 非目标明确
- 关键决策可追踪

### 5. 切分 roadmap

把工作拆成 3~8 个 phase，明确：

- phase 目标
- 输入前提
- 输出物
- 风险点
- 依赖关系

### 6. 初始化状态

在 `STATE.md` 中写明：

- 当前阶段
- 当前焦点
- 最近动作
- 下一步建议

## 约束与注意事项

- 不要在初始化阶段直接给出过细执行方案
- phase 不要切得太碎
- 需求中要明确写出“不做什么”
- 如果是既有系统，应先参考 `gsd-map-codebase`

## 与其他 skills 的衔接关系

- 初始化完成后，通常进入 `gsd-discuss-phase`
- 如果发现是既有系统且背景不清，先转 `gsd-map-codebase`
- 如果需求其实很小，可转 `gsd-fast`
