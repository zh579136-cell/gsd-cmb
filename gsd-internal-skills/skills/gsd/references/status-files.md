# 状态文件说明

## 目标

状态文件的目标不是堆文档，而是让 agent 和协作者能在任意时刻快速回答三类问题：

- 我们要做什么
- 现在做到哪了
- 下一步该做什么

## 建议保留的文件

### `PROJECT.md`

项目级定义文件。

回答：
- 这个项目是什么
- 核心价值是什么
- 有哪些关键约束和关键决策

### `REQUIREMENTS.md`

需求范围文件。

回答：
- 当前要交付什么
- 什么明确不做
- 验收边界在哪里

### `ROADMAP.md`

phase 规划文件。

回答：
- 当前有哪些 phase
- 每个 phase 的目标、输入、输出、依赖是什么

### `STATE.md`

项目状态摘要文件。

回答：
- 当前阶段是什么
- 最近做了什么
- 当前阻塞点是什么
- 下一步是什么

### `CONTEXT.md`

phase 级补充上下文。

回答：
- 这个 phase 的边界和决定是什么
- 有哪些参考资料和约束

### `PLAN.md`

phase 实施计划。

回答：
- 具体要做哪些任务
- 依赖关系是什么
- 每项任务怎么验收

### `TASKS.md`

执行记录。

回答：
- 哪些任务完成了
- 哪些任务阻塞了
- 做了哪些偏离计划的处理

### `VERIFY.md`

验证与回流记录。

回答：
- 验证做了什么
- 通过了什么
- 有哪些问题需要返工

### `CODEBASE_MAP.md`

代码库地图。

回答：
- 代码结构、模块关系、测试方式、风险点是什么

## 使用原则

- `STATE.md` 应保持简洁，方便一眼看清当前状态
- `PLAN.md` 负责行动，`TASKS.md` 负责过程，`VERIFY.md` 负责结果
- 既有系统优先建立 `CODEBASE_MAP.md`
- phase 开始前尽量先有 `CONTEXT.md`
